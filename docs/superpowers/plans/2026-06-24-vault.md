# The Vault Implementation Plan

> **For agentic workers:** execute task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
> Source of truth for design + data models + RLS is the spec: `docs/superpowers/specs/2026-06-24-vault-design.md`.
> This plan sequences that spec into buildable segments. Read the referenced spec section before each task.

**Goal:** Build the Map's Vault (Desire Map reuse + dual-lock Agreements + private/shared Event Log + the
"a decline never discloses" Consent exchange), reachable from the Us-layer Vault row.

**Architecture:** View → `VaultStore` (`@Observable @MainActor`) → services / Supabase edge functions →
SwiftData `@Model`s + Supabase tables. Private/shared split reuses the `desire_ratings` (user-scoped) vs
`desire_matches` (couple-scoped) RLS primitive. Sensitive consent logic is edge-function-mediated.

**Tech Stack:** SwiftUI (Swift 6, iOS 16+), SwiftData, Supabase (Postgres 17 + RLS + Edge Functions), the
existing `SyncManager`/`SyncTask` queue, `EntitlementStore`/`PaywallSheet`.

## File structure

**New Swift (Features/Map/Vault/):**
- `VaultStore.swift` — `@Observable @MainActor`, owns the segment selection + agreements/log/consent state; fetches + calls services.
- `VaultSheet.swift` — the sheet shell: 3-segment control + the three section views.
- `Components/VaultDesireSection.swift` — Desire Map summary + where-you-align + locked/paywall + consent entry.
- `Components/VaultAgreementsSection.swift` — safe word + active list + pending proposals + add.
- `Components/VaultLogSection.swift` — the event-log timeline.
- `EventEntryEditorSheet.swift` — add/edit an event-log entry.
- `AgreementProposeSheet.swift` — propose/add an agreement.

**New models (Core/Models/):** `Agreement.swift`, `EventLogEntry.swift`, `ConsentRequest.swift`
(+ enums `EventMood`/`EventTag` in `Core/Models/Enums/AppEventEnums.swift`). All registered in `SchemaV1.models`.

**New services (Core/Services/):** `AgreementsService.swift`, `EventLogService.swift`, `ConsentService.swift`
(thin Supabase upsert/fetch + edge-function callers, mirroring `DesireSyncService`).

**New backend (supabase/):** `migrations/<ts>_vault.sql` (tables + RLS), `functions/consent-ask/index.ts`,
`functions/consent-respond/index.ts`.

**Modified:** `App/ModelContainer.swift` (register new models), `Features/Map/MapView.swift` (present `VaultSheet`
instead of the placeholder), `Core/Services/SyncManager.swift` (new `SyncTask` taskType cases).

## Segment 2 — Agreements (dual-lock; Swift + backend). Spec §3

**Done (device, two-device):** one partner proposes an agreement or a change, the other accepts, it syncs
and both see it; the proposer cannot self-accept; nothing changes until accepted.

### Task 2.1: Backend migration — `agreements` table + RLS
**Files:** Create `supabase/migrations/<timestamp>_agreements.sql`

- [ ] **Step 1:** Write the SQL (Bryan applies it; do NOT run via MCP):
```sql
create table public.agreements (
  id uuid primary key default gen_random_uuid(),
  couple_id uuid not null references public.couples(id),
  text text not null,
  proposed_text text,
  status text not null default 'active'
    check (status in ('active','pendingCreate','pendingChange','pendingRetire')),
  proposed_by uuid references public.user_profiles(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  is_active boolean not null default true
);
alter table public.agreements enable row level security;
-- couple-scoped read + write (both partners; dual-lock enforced in app logic)
create policy "Partners read agreements" on public.agreements for select to authenticated
  using (couple_id in (select c.id from couples c
    where c.user_a in (select id from user_profiles where auth_id = auth.uid())
       or c.user_b in (select id from user_profiles where auth_id = auth.uid())));
create policy "Partners write agreements" on public.agreements for insert to authenticated
  with check (couple_id in (select c.id from couples c
    where c.user_a in (select id from user_profiles where auth_id = auth.uid())
       or c.user_b in (select id from user_profiles where auth_id = auth.uid())));
create policy "Partners update agreements" on public.agreements for update to authenticated
  using (couple_id in (select c.id from couples c
    where c.user_a in (select id from user_profiles where auth_id = auth.uid())
       or c.user_b in (select id from user_profiles where auth_id = auth.uid())));
```

## Segment 3 — Event Log (private/shared; Swift + backend). Spec §5

**Done (device, two-device):** a private entry is visible only to its author (partner cannot see it); a
shared entry appears for both. Add/edit works; the private/shared toggle is prominent.

### Task 3.1: Backend migration — `event_log_entries` + combined RLS
**Files:** Create `supabase/migrations/<timestamp>_event_log.sql`

- [ ] **Step 1:** Write the SQL (Bryan applies):
```sql
create table public.event_log_entries (
  id uuid primary key default gen_random_uuid(),
  author_id uuid not null references public.user_profiles(id),
  couple_id uuid references public.couples(id),
  date timestamptz not null,
  title text not null,
  note text,
  mood text not null,
  tags jsonb not null default '[]'::jsonb,
  who text,
  visibility text not null check (visibility in ('private','shared')),
  created_at timestamptz not null default now()
);
alter table public.event_log_entries enable row level security;
-- combined: shared rows are couple-visible; private rows are author-only
create policy "Read own private or shared couple entries" on public.event_log_entries for select to authenticated
  using (
    (visibility = 'private' and author_id in (select id from user_profiles where auth_id = auth.uid()))
    or
    (visibility = 'shared' and couple_id in (select c.id from couples c
       where c.user_a in (select id from user_profiles where auth_id = auth.uid())
          or c.user_b in (select id from user_profiles where auth_id = auth.uid())))
  );
create policy "Insert own entries" on public.event_log_entries for insert to authenticated
  with check (author_id in (select id from user_profiles where auth_id = auth.uid()));
create policy "Update own entries" on public.event_log_entries for update to authenticated
  using (author_id in (select id from user_profiles where auth_id = auth.uid()));
```

## Segment 4 — Consent exchange (privacy-critical; Swift + backend). Spec §4

**Done (device, two-device privacy test):** A asks to open a topic; B sees the request in-app and can Open or
decline; on Open both see a neutral discussion card; on decline, A's view is **identical to still-pending**
(the decline-never-discloses test passes). No push.

### Task 4.1: Backend — `consent_requests` + private decline record + RLS
**Files:** Create `supabase/migrations/<timestamp>_consent.sql`

- [ ] **Step 1:** Write the SQL (Bryan applies). `consent_requests` is couple-readable and only ever holds
  `pending`/`opened`; declines are recorded in `consent_declines` readable only by the decliner (service
  role writes both). Full table + RLS per spec §4:
```sql
create table public.consent_requests (
  id uuid primary key default gen_random_uuid(),
  couple_id uuid not null references public.couples(id),
  item_id text not null,
  asker_id uuid not null references public.user_profiles(id),
  status text not null default 'pending' check (status in ('pending','opened')),
  created_at timestamptz not null default now()
);
create table public.consent_declines (   -- responder-private; the asker can never read this
  id uuid primary key default gen_random_uuid(),
  couple_id uuid not null references public.couples(id),
  item_id text not null,
  decliner_id uuid not null references public.user_profiles(id),
  created_at timestamptz not null default now()
);
alter table public.consent_requests enable row level security;
alter table public.consent_declines enable row level security;
create policy "Partners read consent requests" on public.consent_requests for select to authenticated
  using (couple_id in (select c.id from couples c
    where c.user_a in (select id from user_profiles where auth_id = auth.uid())
       or c.user_b in (select id from user_profiles where auth_id = auth.uid())));
create policy "Decliner reads own declines" on public.consent_declines for select to authenticated
  using (decliner_id in (select id from user_profiles where auth_id = auth.uid()));
-- No INSERT/UPDATE policies: writes happen via service-role edge functions only.
```
- [ ] **Step 2:** After apply, `get_advisors(type: security)` clean; manually confirm `consent_declines`
  has NO policy that the asker could satisfy.

### Task 4.2: Edge functions — `consent-ask`, `consent-respond`
**Files:** Create `supabase/functions/consent-ask/index.ts`, `supabase/functions/consent-respond/index.ts`

- [ ] **Step 1:** `consent-ask(item_id)`: service-role; if a `consent_declines` row exists for that
  (couple,item) it returns success WITHOUT re-nudging (silent), else upserts a `pending` `consent_requests`
  row. Returns the same shape regardless, so the asker cannot infer a prior decline.
- [ ] **Step 2:** `consent-respond(request_id, action)`: service-role; `open` sets `status='opened'` and
  records nothing in declines; `decline` deletes/parks the request AND inserts a `consent_declines` row.
  The asker-visible state after a decline is indistinguishable from pending.
- [ ] **Step 3:** Bryan deploys both (`supabase functions deploy`). Note in the handoff that these are
  service-role and must never echo decline state to the caller.

### Task 4.3: ConsentService + VaultStore
**Files:** Create `Vayl/Core/Services/ConsentService.swift`; Modify `VaultStore.swift`

- [ ] **Step 1:** Service: `ask(itemId:) async`, `respond(requestId:open:) async` (call the edge functions);
  `fetchRequests(coupleId:) async -> [ConsentRequestRow]` (reads `consent_requests`, RLS-filtered).
- [ ] **Step 2:** `VaultStore`: `consentRequests`, `loadConsent()`, `ask(itemId:)`, `open(_:)`, `decline(_:)`.
  The asker UI binds to `pending`/`opened` only; there is no "declined" state to bind.
- [ ] **Step 3: Compile.**

### Task 4.4: Consent UI in the Desire section
**Files:** Modify `Vayl/Features/Map/Vault/Components/VaultDesireSection.swift`

- [ ] **Step 1:** Add the "Open a conversation" block: for the current user as asker, rows showing
  ask / pending / opened (+ the neutral discussion card on opened); as responder, incoming requests with
  Open / Not now. Identical-by-construction asker view for pending vs declined. Spec §4.
- [ ] **Step 2: Compile.** Then the two-device decline-never-discloses done-condition (the gating test).
