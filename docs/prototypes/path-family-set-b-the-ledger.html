Set B

The Path — the ledger

The trail is reserved for the one screen whose job is actually spatial — walking your current phase. Everywhere else uses the language that fits the task: the path hub is a plain status list, node editing expands inline instead of opening a sheet, and Support gets a chronological moments feed instead of a read-only map. Each screen is lighter for its own job; the tradeoff is two visual languages to keep coherent instead of one.

01Your paths — a status list

No mini-trails. A dot, a name, a status word. Faster to scan, and it doesn't pretend a locked or paused path deserves the same visual weight as the one you're actually walking.

Where this set is strongest: every screen is sized to its actual job — a scan-list for choosing, a feed for a role that doesn't walk the map, an inline expand instead of a heavier sheet. Likely cheaper to build (list rows and cards are simpler than serpentine SVG geometry) and easier to keep legible at Dynamic Type sizes. Where it strains: two visual languages (trail + ledger) in one feature means more design surface to keep coherent, and Support's "moments feed" is a bigger structural swing — worth pressure-testing whether it feels like a real feature or a consolation screen.
